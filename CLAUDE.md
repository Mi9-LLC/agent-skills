# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository purpose

This is the **Mi9 LLC public catalog of Claude Code Agent Skills**, hosted on GitHub at `github.com/Mi9-LLC/agent-skills`. Teammates and external users install individual skills with:

```
npx skills add https://github.com/Mi9-LLC/agent-skills --skill <skill-name>
```

It is **not** application code. There is no build, test, or lint pipeline — the entire repo is Markdown skill definitions. Do not invent commands; there are none to run.

## Layout

Flat skills layout, matching the canonical [`anthropics/skills`](https://github.com/anthropics/skills) example:

- `skills/<name>/SKILL.md` — the **skill definition** (YAML frontmatter with `name`, `description`, `allowed-tools` + Markdown body that's loaded into the model when the skill triggers).
- `skills/<name>/references/` — on-demand reference docs the skill reads itself when it needs the detail. Keep these out of `SKILL.md` so the always-loaded body stays small.

There are no plugin manifests, no `.claude-plugin/` directory, and no top-level catalog file. Adding any of those would mix two distribution mechanisms — see "Distribution mechanism" below for why we don't.

Declares `disallowed-tools: Edit, NotebookEdit` (drops those tools while active; a per-turn guard that clears on the next message). Runs a bundled zero-dependency Node script (`scripts/extract-sonar-issues.mjs`, built-in `fetch`); **read-only against the Sonar API** — it queries results, never triggers a scan and never modifies the repo (writes only when `--out <file>` is passed). Hard constraint: **never changes runtime or wire behavior** — code-quality smells only, not bug fixes; genuine bugs are surfaced, not force-fixed. Structural refactors on untested code are **characterization-tests-first** (lock current output with a snapshot, refactor to byte-identical output, keep the tests as regression coverage). Recipes are JS/TS today; the workflow is language-agnostic. Calls `sonar-issue-check`'s script to fetch findings when installed, else accepts a pasted list. `allowed-tools: Read, Edit, Write, Bash, Grep, Glob`. Ships `references/`: per-rule fix recipes (`rule-fixes.md`) and the complexity-refactor playbook (`refactor-playbook.md`). **Does not commit or push unless asked.**

Decides everything from the **built `dist`** via a bundled zero-dependency Node analyzer (`scripts/analyze-initial-load.mjs` — computes the entry's static-import closure, ranks initial-load chunks, flags vendor-looking deferral candidates), then diagnoses the leak (eager import / un-tree-shaken barrel re-export / `manualChunks` hoist), applies the matching fix **on approval**, and verifies against rebuilt artifacts. **Vite/Rollup/Rolldown only** (not Next.js/Webpack). `allowed-tools: Bash, Read, Edit, Write, Grep, Glob`. Ships `references/` (`diagnosis-and-fixes.md`, `verification.md`).

Ships `references/` (interview scripts + an annotated one-shot example) and `templates/`. Vendored from `ragnar-pwninskjold/tech-snacks` (MIT, © 2026 Sean Kochel); adapted to be CLAUDE.md-focused and shell-free. **Does not edit an existing CLAUDE.md or write to the repo root.**

Four phases: root-cause investigation → pattern/context analysis → single tested hypothesis → fix the cause behind a failing test. **Iron law: no fix without root-cause investigation first; after 3 failed fixes, stop and question the architecture rather than trying a 4th.** Decoupled from the superpowers hook/dispatcher/sibling skills; **Windows-clean** (no unix-only shell to run); examples for **.NET/xUnit + TS/Vitest**. Ships `references/` (root-cause-tracing, defense-in-depth, condition-based-waiting). `allowed-tools: Read, Grep, Glob, Bash, Edit, Write`.

Decoupled from the superpowers hook/dispatcher; examples for **.NET/xUnit + TS/Vitest** with Windows-friendly commands. - **`session-handoff`** (`skills/session-handoff/`) — Creates and resumes handoff documents for transferring work between AI agent sessions, so a fresh agent can continue with zero ambiguity. CREATE mode scaffolds a handoff via `scripts/create_handoff.py` (pre-fills timestamp, git branch, recent commits, modified files; supports `--continues-from` to chain handoffs), then validates it with `scripts/validate_handoff.py` (blocks finalizing if secrets are detected — API keys, passwords, tokens, private keys, connection strings — or the quality score is below 70). RESUME mode lists handoffs (`scripts/list_handoffs.py`) and grades staleness FRESH → SLIGHTLY_STALE → STALE → VERY_STALE from time elapsed, commits, and file/branch drift (`scripts/check_staleness.py`) before loading. Bundled scripts are Python, stdlib only (Python 3.9+, nothing to `pip install`). **Writes only under `.claude/handoffs/`** (`YYYY-MM-DD-HHMMSS-<slug>.md`). Declares no `allowed-tools`/`disallowed-tools` (unrestricted). Ships `references/` (`handoff-template.md`, `resume-checklist.md`) and its own `README.md`.
- **`retro`** (`skills/retro/`) — Git-log engineering retrospective for the current repo. A bundled zero-dependency Node script (`scripts/git-retro.mjs`, Node >= 18) computes every metric deterministically (commits, contributors, LOC, test ratio, per-author 45-minute-gap work sessions, hourly histogram, commit-type mix, churn hotspots, approximate PR count/size buckets, focus score, ship-of-the-window, streaks, AI-assisted-commit share via `Co-Authored-By` trailers) and emits JSON; the model only writes the narrative on top of it and may never state a number absent from that JSON. Flags: `--window 7d|24h|Nd|Nh|Nw` (default 7-day window), `--compare` (prior same-length window computed live from git, plus deltas), `--base <ref>`, `--no-fetch`. Honesty guards in the JSON (`zeroCommits`, `staleBase`, `fetchFailed`, `noRemote`, `detachedHead`, `shallowClone`) must be carried into the narrative verbatim — a zero-commit window is reported as exactly that, never padded. Terminal-only by default; `--save` writes a JSON snapshot to `docs/retros/` and the model then writes `docs/retros/<YYYY-MM-DD>-retro.md`. **Never commits or pushes.** Per-author output is stats-only (leaderboard + a personal deep-dive for the runner) with no individual praise/critique — improvement suggestions are team-level only, and it refuses performance-review/HR asks. `allowed-tools: Bash, Read, Write`; `disallowed-tools: Edit, NotebookEdit`. Adapted from `garrytan/gstack` (MIT, © 2026 Garry Tan); gstack-state integrations (learnings, Greptile, telemetry, global mode) dropped.
- **`verify-frontend-change`** (`skills/verify-frontend-change/`) — Browser-verification loop that forbids calling a frontend/UI change done from a successful edit, green build, or HMR reload alone. Five steps driven through the **Chrome DevTools MCP server** (required; if absent the skill stops and reports the change *unverified*): start/reuse the dev server (background task; URL taken from the server's own output, never assumed) → open the affected route → interact with the changed behavior itself → console gate (zero new errors; unattributable errors count as new, pre-existing ones reported, never silent) → performance trace (fails on change-attributable regressions; dev-mode-numbers caveat). **Any failure → fix → rerun from Step 1**, never resume mid-loop; after 3 failed fix cycles on the same step, stop and question the approach (points to `systematic-debugging`). Exactly two end states: Verified-with-evidence or Blocked/unverified — no "probably fine". Stops only dev servers it started. Declares no `allowed-tools`/`disallowed-tools` (unrestricted — the browser tools it needs are the user's own `chrome-devtools` MCP registration). Ships `references/` (`chrome-devtools-mcp.md` tool cheatsheet, `dev-server-playbook.md`).
- **`health`** (`skills/health/`) — Local quality-gate dashboard for the current repo. A bundled zero-dependency Node script (`scripts/check-health.mjs`, Node >= 18) runs the project's own gates (typecheck / lint / test / dead-code / shell lint), parses their output, scores each 0–10 against a weighted rubric (typecheck 25 / lint 20 / test 30 / deadcode 15 / shell 10, renormalized over whichever gates ran), and computes the weighted composite; the model only narrates that JSON and may never state or estimate a number absent from it. Config lives in `.claude/health.json`, written by the model only after the user confirms the detected stack on first run (`--detect-only` prints the proposed config; AskUserQuestion confirm; stdout written verbatim). A category whose tool isn't installed is skipped and its weight redistributes — skipped ≠ failed, never scored. Flags: `--detect-only`, `--config <path>`, `--only <cats>`, `--save [dir]` (opt-in history append to `docs/health/history.jsonl`; trend — delta, direction, per-category regressions — is read from that file automatically whenever it exists). Honesty guards carried into the dashboard verbatim: `noToolsDetected` (composite `null`, honest empty dashboard), `notGitRepo`, `dirtyWorkingTree`, `anyTimeout` (hung gate scores 0), `anyParseFallback` (exit-code-only scoring flagged as coarse), `firstRun`. Never fixes issues, never modifies the repo (writes only the `--save` history line; the model may Write a dated `docs/health/<YYYY-MM-DD>-health.md` report only on an explicit ask). **Never commits or pushes.** `allowed-tools: Bash, Write`; `disallowed-tools: Edit, NotebookEdit`. Adapted from `garrytan/gstack` (MIT, © 2026 Garry Tan); rebuilt — deterministic script replaces model-computed scoring; gbrain and `~/.gstack` state dropped; config moved to `.claude/health.json`; history opt-in and in-repo.
- **`plan-eng-review`** (`skills/plan-eng-review/`) — Pre-implementation gate that reviews a **written implementation plan** (plan-mode draft, a file under `docs/plans/`, or a pasted plan) before any code is written: scope challenge → what-already-exists reuse check → four dimensions (architecture, code quality of the planned code, tests — the heaviest, performance) → verdict (`APPROVED` / `APPROVED WITH CHANGES` / `NEEDS REVISION`, bound by a decision table: any CRITICAL GAP or unresolved decision ⇒ NEEDS REVISION; non-empty Required plan changes ⇒ at most APPROVED WITH CHANGES). **Iron law: no finding without evidence** — presence findings quote the plan or a verified `file:line`; absence findings quote the obligation-creating plan text plus the verified negative search; neither ⇒ at most a low-confidence appendix note. **REGRESSION RULE (non-negotiable): plan modifies existing behavior with no covering test → regression test enters Required plan changes, no question asked.** Max 8 findings per dimension, severity-ranked, confidence 1–10. Genuine scope/design forks go to the user via AskUserQuestion **batched per section** (never one-interrupt-per-finding); unanswered forks land under `UNRESOLVED DECISIONS:`, never silently defaulted; answered ones are recorded in a Decisions block and carried forward on re-runs. **The only mutation it ever performs** is replacing/appending the plan file's `## ENG REVIEW REPORT` section via a whole-file Write-splice (H2 boundaries matched outside fenced code blocks, trailing-`\r` tolerant; report interior `###`-only; every non-report byte preserved); no plan file on disk → terminal-only report, zero writes. Optional outside voice (one subagent prompted to refute the verdict) on explicit ask only. **Never implements the plan; never commits or pushes.** Catalog position: `new-feature` (design) → plan mode → **`plan-eng-review` (gate)** → `convert-plan-to-feature` (decomposition). `allowed-tools: Read, Grep, Glob, Bash, Write, Agent` (Bash read-only; Agent only for the optional outside-voice subagent); `disallowed-tools: Edit, NotebookEdit`. Ships `references/review-dimensions.md` (four checklists, coverage legend, severity/confidence tables, verdict rules, report skeleton). Adapted from `/plan-eng-review` in `garrytan/gstack` (MIT, © 2026 Garry Tan); per-finding question gates → per-section batching, Codex outside-voice → optional on-request Claude subagent, test-plan/tasks artifacts folded into the single in-plan report, iron law extended to absence findings, gstack state (`~/.gstack`, `bin/` loggers, brain, TODOS.md, readiness dashboard, review chaining, Lake Score) dropped.
- **`document-generate`** (`skills/document-generate/`) — Writes human-facing documentation files — Diataxis tutorials, how-to guides, reference pages, explanations — for a named feature, module, or whole project, grounded in end-to-end codebase archaeology (implementation + tests read first) before any prose. Classifies each entity into the four Diataxis quadrants via a 7-row decision matrix, then **never writes a doc file before the partition plan is approved** via an always-on AskUserQuestion gate — an unanswered or blocked gate means nothing is written. **Iron law:** every example is executed (ran side-effect-free this session), traced (every identifier read at a real `file:line` this session), or illustrative (shape traced, explicitly labeled) — in that preference order; an example meeting none of these gets rewritten or cut. Collision policy: a planned file that already exists becomes an `extend` row — read fully, merged under its structure, authored prose preserved **except where it contradicts the code** (code wins every factual/API claim; every correction reported). Docs home resolves target-local convention → repo `docs/` → detected doc-framework config (Docusaurus/MkDocs/VitePress/Nextra, sidebar updated) → new root `docs/`. `allowed-tools: Read, Grep, Glob, Bash, Write, Edit` — **no `disallowed-tools`**: the skill needs Edit and Bash to do its job, so its never-rules (**never edits `CLAUDE.md`/`AGENTS.md`; never commits or pushes**) are workflow discipline rather than a tool-pool restriction, and the body says so explicitly. Ships `references/quadrant-templates.md` (four templates, per-quadrant rules, anti-mixing table, example-verification techniques, doc-framework identification table, collision-policy detail). Adapted from `/document-generate` in `garrytan/gstack` (MIT, © 2026 Garry Tan); the commit/push/PR-body-update tail is dropped (this skill never commits), the `gstack-redact` binary becomes the placeholder-credentials rule, the confirm-only-above-5-docs threshold becomes the always-on partition-plan approval gate, upstream's inline-summaries-plus-standalone-files default narrows to standalone files plus minimal link lines, and the gstack machinery is dropped.
- **`stdlib-first`** (`skills/stdlib-first/`) — Behavioral reuse-before-build discipline for new TypeScript/Node and C#/.NET code: a strict 4-rung ladder (built-in/standard-library API → C#-only first-party `Microsoft.Extensions.*` → widely-adopted library the project already uses → custom implementation last), plus precise types (no `any`, no `object`/`dynamic`), specific error classes over bare `Error`/`Exception`, C# async with `CancellationToken` and no sync-over-async, and short doc comments on non-obvious members. **Always asks before adding a dependency (AskUserQuestion); produces no files — changes only the code Claude writes.** Description carved to the two decision moments (writing a new utility/helper/service; adding/choosing a dependency); do-NOT covers code review, debugging existing behavior, trivial edits, and other languages. Canonical copy of the `## Coding standards` block checked into Mi9 repo `CLAUDE.md` files (their `<!-- source -->` comments point here) and the bootstrap for future repos. Declares no `allowed-tools`/`disallowed-tools` (guides code-writing; must not restrict tools). No `references/` (body is small).
- **`repo-change-summary`** (`skills/repo-change-summary/`) — Per-month change totals for a git repo across **all branches**: lines added/deleted, total lines changed (churn, not net), distinct files modified, file-touches, commits, PRs merged, authors — each commit counted once, merges excluded from the line/file/commit counts. A bundled POSIX-shell script (`scripts/summary.sh`, needs only `git` + a POSIX shell; kept LF via `.gitattributes`) computes everything deterministically, prints a Markdown table the model relays **verbatim**, and writes a self-contained HTML report (`YYYY-MM-DD-HHMM-repo-change-summary-<month>.html`) that opens in the browser (`--no-open` to suppress). Defaults to the current month and current repo; `--month YYYY-MM`, `--repo PATH`, `--out DIR`, `--no-fetch`. Scope is local + remote-tracking branches (stash entries and tag-only commits excluded); authors respect `.mailmap`. PR count = distinct PR numbers from platform merge messages (GitHub subjects, Bitbucket "(pull request #N)" subjects incl. its squash merges, GitLab "See merge request <path>!N" body lines) — reverts, body mentions, and re-merges don't inflate it; markerless squash/rebase merges aren't counted. Companion `scripts/multi-summary.sh --group NAME` rolls up a **named group of repos** into one combined Markdown + HTML report (rollup totals row + per-repo tables; group authors = distinct people across the group, never the column sum; fetch-failed repos marked `*`); groups are machine-local `.list` files under `~/.claude/repo-change-summary-groups/`, one repo path per line — deliberately outside the skill because clone paths differ per machine. `--per-author` appends a per-developer **activity** table (sorted by total changed ascending like the rollup, explicitly activity-not-performance, single-dominating-file footnotes; PRs-authored via Bitbucket API through `scripts/pr-authors.py`, Python stdlib, creds from `git credential fill` as `<git user.email>:<API token>` with `BITBUCKET_EMAIL` override — column omitted with a note when unavailable; bot identities excluded from the table and footnoted — built-in `commits-noreply@bitbucket.org`, extend via `<groups-dir>/bot-emails.list` — rollup keeps every commit). The combined HTML embeds inline-SVG bar charts (zero JS/CDN; palette CVD-validated): lines changed by repo, plus per-developer lines + PRs charts under `--per-author`. Both scripts also support optional **PDF-email delivery** (`--email`, `--to LIST`, `--subject STR`, `--email-dry-run`, `--env-file PATH`, `--mailmap PATH`) via the bundled `scripts/send-report.py` (Python 3.9+, stdlib only): **preview-first** — `--email-dry-run` resolves every recipient (literal email, or a tiered case-insensitive `.mailmap` name match) and renders the HTML report to PDF via the locally-installed headless browser (Chrome preferred, Edge fallback — an Oct-2025 Edge-141 `--print-to-pdf` regression is why) without sending anything; only a confirmed re-run without that flag actually sends, over SMTP (AWS SES, credentials from `~/.claude/repo-change-summary.env`) with a modern email-safe HTML body plus the PDF attached. `allowed-tools: Bash`; pins `model: claude-sonnet-5` (the run is script-driven — deterministic scripts do all computation, so the mid-tier model suffices). Ships `references/emailing.md`.

## Contributing

1. Create or edit a directory under `skills/<skill-name>/`. Include a `SKILL.md`; put long-form reference docs under `references/`.
2. **When adding (or renaming/removing) a skill, update ALL catalog docs in the same change — this is mandatory, not optional:**
   - `README.md` — add a row to the **Skills at a glance** table *and* a full per-skill section (mirror the existing format: *what it does*, *use it for*, *triggers on*, *what it does not do* / *what it produces*, *install*, *full definition* link). The README is how consumers discover skills — an undocumented skill is effectively unshipped.
   - This `CLAUDE.md` — add a bullet to the **Current skills** list above.
   - Keep the table-row count, the `## ` section count, and the `skills/` directory count in agreement.
3. Open a PR against `main`. Teammates pick up changes on their next `npx skills add … --skill <name>`.

There are no versions to bump and no catalogs to update — `npx skills add` always pulls the current state of the branch it points at.

## Distribution mechanism

This repo distributes **only** via `npx skills`. The earlier Bitbucket-hosted `mi9-plugins` marketplace (using `.claude-plugin/marketplace.json` + `/plugin install`) was retired because the Claude Code TUI's `/plugin install` flow hit a "source type not supported" bug that blocked teammates from installing through the UI. The skills-only model bypasses that flow entirely.

If we ever need to ship non-skill components (agents, hooks, MCP servers, LSP) from the same repo, the two options are:

1. Add a `.claude-plugin/marketplace.json` back alongside the flat `skills/` tree (dual-mode — `anthropics/skills` itself does this), or
2. Split into a second repo.

Not in scope today; flagged for future reconsideration.
